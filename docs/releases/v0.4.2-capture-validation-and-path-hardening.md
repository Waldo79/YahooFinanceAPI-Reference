# Capture Validation and Path Hardening — v0.4.2

## Reason for this release

A live v0.4.1 capture successfully returned matching HTTP 200 results for all 16 representative symbols. Review confirmed that the raw files, hashes, metadata sidecars, normalized output, request order, and session redaction were internally consistent.

The review also exposed two portability weaknesses:

1. the manifest stored complete local Windows paths, including the contributor's username and directory layout; and
2. the default `captures/local` destination depended on the Command Prompt's current directory.

v0.4.2 corrects both weaknesses and adds a formal completed-run validator.

## Fixed output destination

The default output directory is derived from the script's repository location:

```text
<repository-root>/captures/local
```

It no longer depends on where the command was launched. An explicit `--outdir` still overrides the default.

## Portable manifest paths

Paths within the repository are stored using forward-slash repository-relative references, for example:

```text
tools/capture-utility/symbols.csv
data/master_field_database.csv
```

An input located outside the repository is recorded by filename only and marked with scope `external`. This avoids publishing local usernames and directory layouts.

## Run validator

The command:

```text
py tools\capture-utility\yahoo_capture.py --validate-run captures\local\<run-folder>
```

checks:

- manifest JSON structure and completion;
- folder name versus run ID;
- request count, order, sequence, and symbols;
- summary classifications;
- manifest entries versus metadata sidecars;
- referenced file existence and safe in-run paths;
- filename sequence prefixes;
- raw SHA-256 and response byte counts;
- JSON parseability when JSON is expected;
- missing, duplicate, and unreferenced files;
- unredacted crumb, cookie, and authorization values;
- local absolute paths; and
- JSON paths known or unmapped in the master field database.

## Validation reports

Each validation writes:

- `run-validation.json` — machine-readable results conforming to `schemas/run-validation.schema.json`;
- `run-validation.txt` — a concise human-readable review report.

Status values are `PASS`, `PASS_WITH_WARNINGS`, and `FAIL`.

Unmapped paths are informational. The validator does not modify raw evidence or automatically update the master field database.

## Progress display

Capture now displays one result line after each symbol:

```text
[01/16] AAPL ... HTTP 200 SUCCESS_RESULT_RETURNED
```

This makes long runs easier to follow without exposing cookie or crumb values.

## Compatibility

- Python 3.10 or newer
- Python standard library only for capture and validation
- `pytest` is optional and used only for development tests
- Existing v0.4.0 and v0.4.1 run folders can be validated
- Legacy runs with complete local manifest paths receive warnings rather than hash or structural failures

## Evidence rule

Validation reports are derived review artifacts. They do not alter or replace the raw Yahoo response, its SHA-256 value, metadata sidecar, or original run manifest.
